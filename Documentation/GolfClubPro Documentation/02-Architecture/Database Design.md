# Purpose

The Database Design defines how GolfClubPro stores player, course, round, shot, recommendation and analytics data.

The purpose of this document is to ensure that persistence supports:

- Offline play
- Reliable round capture
- Future CloudKit synchronisation
- AI learning
- Post-round analytics
- Long-term player history

---

# Persistence Strategy

GolfClubPro will use an Apple-first persistence strategy.

## Local Persistence

SwiftData will be used for local storage on iPhone and Apple Watch where appropriate.

## Cloud Synchronisation

CloudKit will be used for optional synchronisation and backup.

## Offline First

The application must support a complete round without network access.

Network availability must not be required for:

- Starting a round
- Recording shots
- Viewing current hole information
- Recording club use
- Completing a round

---

# Persistence Boundary

GolfCore should remain independent of SwiftData and CloudKit.

GolfCore defines the domain and data models.

The app layer or persistence adapter is responsible for mapping GolfCore models into SwiftData models.

This preserves testability and avoids coupling the core domain to a storage framework.

# Core Persistence Entities

## PlayerRecord

Represents the locally stored player profile.

Fields may include:

- id
- dotGolfMemberID
- name
- handicapIndex
- createdAt
- updatedAt

## ClubRecord

Represents one club in a player's equipment profile.

Fields may include:

- id
- playerID
- name
- clubType
- loftDegrees
- averageCarryMeters
- createdAt
- updatedAt

## CourseRecord

Represents a playable golf course.

Fields may include:

- id
- name
- region
- country
- courseVersion
- createdAt
- updatedAt

## HoleRecord

Represents one hole within a course.

Fields may include:

- id
- courseID
- number
- par
- strokeIndex
- lengthMeters

## RoundRecord

Represents one round played by a player.

Fields may include:

- id
- playerID
- courseID
- status
- startedAt
- completedAt
- selectedTeeSet

## ShotRecord

Represents one recorded shot.

Fields may include:

- id
- roundID
- holeID
- clubID
- timestamp
- distanceMeters
- lie
- weatherContextID
- startLatitude
- startLongitude
- endLatitude
- endLongitude

## RecommendationRecord

Represents one recommendation generated by the AI Caddy.

Fields may include:

- id
- roundID
- holeID
- suggestedClubID
- confidenceScore
- explanation
- createdAt

# Append-Only Shot History

Shot history should be treated as append-only where practical.

This means:

- Completed shots should not be silently overwritten.
- Corrections should be recorded explicitly.
- Historical shot records should remain available for analytics.
- AI learning should use the corrected final state while preserving audit history where useful.

This supports explainability, debugging and future coaching insights.

---

# Sync Strategy

CloudKit synchronisation should be introduced after local persistence is stable.

The sync strategy should support:

- Offline creation of rounds
- Later cloud upload
- Multi-device availability
- Conflict detection
- Conflict resolution

## Conflict Resolution

Potential rules:

- Newer profile metadata may overwrite older metadata.
- Shot records should prefer append-only merge.
- Round completion status should use latest timestamp.
- Course data should be versioned.

---

# Watch Storage Strategy

The Apple Watch should store only the minimum data required during play.

Watch-local data may include:

- Active round summary
- Current hole
- Player clubs
- Recent shots
- Current recommendation
- Cached course data required for the round

The iPhone should remain the primary long-term data store.

---

# Course Data Versioning

Course data should be versioned because course layouts, tee markers, stroke indexes and local rules can change.

Each course record should eventually support:

- Course version
- Effective date
- Source
- Last verified date

---

# Privacy and Security

GolfClubPro stores personal performance data.

The database design should support:

- User ownership of data
- Local-first storage
- Secure cloud sync
- No unnecessary third-party sharing
- Clear deletion/export strategy

---

# Migration Strategy

Database migrations should be planned before App Store release.

Migration principles:

- Preserve player history
- Preserve shot history
- Avoid destructive migrations
- Test migrations using sample data
- Maintain backward compatibility where practical
